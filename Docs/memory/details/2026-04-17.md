### 🟡 新问题 A（次要）：CPU 下采样用最近邻，不是双线性

```520:528:Source/LookScopes/Private/AIColorGrader.cpp
	for (int32 y = 0; y < DstH; y++)
	{
		const int32 sy = y * SrcH / DstH;
```

这是**最近邻采样**，跟训练时 `PIL.Image.resize(BILINEAR)` 或 OpenCV 的 `INTER_LINEAR` 不一致，会改变图像低频统计（颜色均值轻微偏移）。

### 🔴 新问题 B（高嫌疑）：`PreExposure` 的除法方向可能是反的

```91:92:Source/LookScopes/Private/AIGradingViewExtension.cpp
	const float EyeExp = View.GetLastEyeAdaptationExposure();
	const float PreExposure = EyeExp > 0.0f ? EyeExp : 1.0f;
```

```182:182:Source/LookScopes/Private/AIGradingViewExtension.cpp
	DispatchDown(DownRDG, AI_SIZE, AI_SIZE, 1.0f / PreExposure);
```

shader 里做的是 `Color *= 1/PreExposure`，意图是"撤销 EyeAdaptation 乘法"。

**但在 `MotionBlur` pass 之后拿到的 SceneColor，EyeAdaptation 还没被乘进去**（EyeAdaptation 是在 Tonemap pass 里应用的，MotionBlur 在 Tonemap 之前）。如果这个理解对，那这行代码是：

- 当前：`Color *= 1/EyeExp`（错误地把还没乘的东西除掉）
- 应该：`Color *= 1.0`（不动）或者 `Color *= EyeExp`（主动应用眼适应）

**症状**：在明亮场景（`EyeExp > 1`），画面被除得变暗；在暗场景（`EyeExp < 1`），画面被除得变亮。模型看到的亮度分布**跟实际画面观感反着**，于是预测的 LUT 朝反方向调。

这**正好能解释"颜色调得不准"**：你在 UE 看到的画面亮度正常，但模型看到的是被亮度反向调过的，它预测的 LUT 会把"本来该压暗的地方往亮调"。

### 🔴 新问题 C（高嫌疑）：训练集 vs 游戏画面的分布不一致

即使上面 B 修好了，还有个更深层问题：

- **训练集 FiveK**：专业摄影师用相机拍、ISP 处理过的消费级 sRGB 照片。分布特征：中间调偏暗、高对比、饱和度自然
- **UE 运行时**：Linear HDR → ACES tonemap → sRGB。ACES 的特点是**中间调偏亮、低对比、高光软 knee**

两个"sRGB 0-255"在**亮度/对比度/饱和度分布**上差距非常大。Classifier 是小 CNN，它对输入分布敏感。在 FiveK 分布上学到的"看到偏暗的照片就加饱和度"，到 ACES 后的游戏画面上就变成"看到偏亮的画面还加饱和度"，结果过饱和。

这是典型的**训练-推理 domain shift**，是比模型本身更根本的问题。

`.cube` 批量生成的核心缺陷是 **LUT 是函数 `f: RGB → RGB`**，而电影级调色往往依赖**上下文**（同一绿色在草地和在皮肤上处理应该不同）。PST50 的配对是真实的"参考图 → 调好色的图"，隐含了上下文信息，能让你的 `Classifier` 学到更敏感的条件分布。

### 2.2 它如何接进你现有的 `train.py`

你当前的 `FiveKDataset` 结构非常干净 —— 只要新建一个并列的 Dataset 类即可。伪代码如下（不是让你马上写，是给你看改动范围）：

```python
class PST50Dataset(Dataset):
    """PST50: photorealistic style transfer pairs."""
    def __init__(self, root, mode="train", crop_size=256):
        self.pairs = _load_pst50_manifest(root, mode)
        self.crop_size = crop_size
        self.mode = mode

    def __getitem__(self, idx):
        input_path, target_path = self.pairs[idx]
        img_input  = Image.open(input_path).convert('RGB')
        img_target = Image.open(target_path).convert('RGB')
        return self._same_aug_pipeline(img_input, img_target)  # 复用 FiveK 那套 crop+resize+hflip
```

然后在 `main()` 里用 `ConcatDataset([FiveKDataset(...), PST50Dataset(...)])` 一行拼起来 —— **网络、loss、训练循环完全不改**。这就是 Image-Adaptive-3DLUT 架构的优势：它把"学什么风格"的问题交给数据，网络只管拟合。

### 2.4 一个细节陷阱要提醒

PST50 的输入/目标分辨率可能不一致（很多 HF 数据集用同一张图做 input 和 target 只是调色不同，但也有 crop 不一致的）。你现在 `FiveKDataset` 里的 `RandomCrop.get_params` 对 input/target 用同一组 `(i,j,h,w)` —— 这**要求两张图尺寸相同**。接 PST50 前必须先 `assert img_input.size == img_target.size`，如果不是就要先 resize 到相同尺寸再 crop。这是踩坑点，别忘了。

**"用 SA-LUT 模型" ≠ "用 SA-LUT 数据"**
