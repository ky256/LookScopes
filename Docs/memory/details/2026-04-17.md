## 一个需要知道的边界

**如果那一轮对话里同时还做了实质工作**，比如"把这个 bug 修一下顺便提交"，那 `git checkout` 会把"修 bug 的讨论"也一起丢掉。  
→ 解决办法：养成"单独一轮纯提交"的习惯，即只说"提交一下"，不掺杂别的事。这本来就是推荐的 git 工作流（atomic commit），顺便带来 memory 干净的好处。

## 自动化建议

手动执行 `git checkout HEAD -- .pending.md` 麻烦。三个自动化方式，你挑：

**方式 A：IDE Task / 快捷键**  
在 VS Code/Cursor 里绑一个键或一个 task runner 按钮，一键执行。零 git 经验最友好。

**方式 B：在提交脚本里自带清洗**  
写个 `infra/memory/commit_and_clean.ps1`：
```powershell
git add .              # stage 本轮所有改动
git commit -m $args[0] # 用你传的 message
git push
# —— 这里等待 hook 异步写完 .pending.md（sleep 1s）——
Start-Sleep -Seconds 1
git checkout HEAD -- Docs/memory/details/.pending.md
```
你之后说"用 commit 脚本提交 xxx"，AI 调脚本，一次搞定。

**方式 C：post-commit 钩子**  
`.git/hooks/post-commit` 自动 `git checkout HEAD -- .pending.md`。问题是：post-commit 在 commit 完成的瞬间就跑，那时候 stop hook 还没触发（stop hook 是 AI 回复结束才跑），所以 post-commit 跑得太早，checkout 什么都没清到。**不适用**。

---

我的推荐：**方式 A**，最轻量。


 I see—the user is explaining that the local pending record only gets populated after they ask me to commit files, and it captures the details of my commit process. They can revert that specific record to surface errors from before the commit attempt. So the actual sequence is: the commit happens during AI's response, then the hook fires after the response ends and appends that entire conversation turn to `.pending.md`. The user's solution is to run `git checkout HEAD -- .pending.md` to discard those post-commit additions and restore the file to its last committed state. They're willing to lose that specific turn about the commit instruction itself since it's meta-work rather than actual project work, and it's recoverable from git history anyway.

Now I'm evaluating whether this approach actually works. The main advantage is it achieves a clean working tree without needing `--skip-worktree` or changing how the file syncs across machines. The tradeoff is it requires manual action after each commit, and there's a potential edge case where substantive content added between commit-prep and commit-done could get lost. But in practice that shouldn't happen if the user is committing when they ask to commit. The real question is whether the hook will re-append the commit turn on the next response, or if it will correctly start from the new baseline.

Looking at how `detail_collector.py` tracks state, it seems to use a checkpoint system that remembers the last processed turn index. So when I run `git checkout HEAD -- .pending.md` to discard the appended content, the state still knows "I've processed up to turn N", meaning the next hook run won't re-append that discarded turn. The content is permanently gone from the file, though it remains in git history. This is exactly what the user wants. I could even automate this with a wrapper script that commits, pushes, then reverts the file to its committed state before the hook runs again.

The cleanest flow is: AI commits and pushes, the hook fires and appends to `.pending.md`, then the user manually runs `git checkout HEAD -- .pending.md` to clean up. Alternatively, this could be bound to a VS Code keybinding for convenience. The main caveat is that the "lost turn" is the commit instruction itself—typically low-value metadata—so it's a good candidate to sacrifice. If the user does substantive work in the same turn as the commit, they'd lose that too, but that's easy to avoid in practice.
