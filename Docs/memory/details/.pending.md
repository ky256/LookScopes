### 🟡 新问题 A（次要）：CPU 下采样用最近邻，不是双线性

```520:528:Source/LookScopes/Private/AIColorGrader.cpp
	for (int32 y = 0; y < DstH; y++)
```

这是**最近邻采样**，跟训练时 `PIL.Image.resize(BILINEAR)` 或 OpenCV 的 `INTER_LINEAR` 不一致，会改变图像低频统计（颜色均值轻微偏移）。

### 🔴 新问题 B（高嫌疑）：`PreExposure` 的除法方向可能是反的

```91:92:Source/LookScopes/Private/AIGradingViewExtension.cpp
	const float EyeExp = View.GetLastEyeAdaptationExposure();
```


shader 里做的是 `Color *= 1/PreExposure`，意图是"撤销 EyeAdaptation 乘法"。

**但在 `MotionBlur` pass 之后拿到的 SceneColor，EyeAdaptation 还没被乘进去**（EyeAdaptation 是在 Tonemap pass 里应用的，MotionBlur 在 Tonemap 之前）。如果这个理解对，那这行代码是：

- 当前：`Color *= 1/EyeExp`（错误地把还没乘的东西除掉）
- 应该：`Color *= 1.0`（不动）或者 `Color *= EyeExp`（主动应用眼适应）

**症状**：在明亮场景（`EyeExp > 1`），画面被除得变暗；在暗场景（`EyeExp < 1`），画面被除得变亮。模型看到的亮度分布**跟实际画面观感反着**，于是预测的 LUT 朝反方向调。

这**正好能解释"颜色调得不准"**：你在 UE 看到的画面亮度正常，但模型看到的是被亮度反向调过的，它预测的 LUT 会把"本来该压暗的地方往亮调"。

### 🔴 新问题 C（高嫌疑）：训练集 vs 游戏画面的分布不一致

即使上面 B 修好了，还有个更深层问题：

- **训练集 FiveK**：专业摄影师用相机拍、ISP 处理过的消费级 sRGB 照片。分布特征：中间调偏暗、高对比、饱和度自然
- **UE 运行时**：Linear HDR → ACES tonemap → sRGB。ACES 的特点是**中间调偏亮、低对比、高光软 knee**

两个"sRGB 0-255"在**亮度/对比度/饱和度分布**上差距非常大。Classifier 是小 CNN，它对输入分布敏感。在 FiveK 分布上学到的"看到偏暗的照片就加饱和度"，到 ACES 后的游戏画面上就变成"看到偏亮的画面还加饱和度"，结果过饱和。

这是典型的**训练-推理 domain shift**，是比模型本身更根本的问题。

**整体定位**：给开发机 AI 的**可执行诊断工单**，不是给人读的分析报告。它假设读者不知道本次对话背景，独立可读。

**核心章节**：

1. **任务说明** — 写清楚训练机已经做了什么（静态审查）、要求开发机做什么（动态验证）、执行原则（先证据后改码）。

2. **4 项按优先级排好的问题** — 每项按 5 段式：
   - 代码位置（带行号和文件路径的 CODE REFERENCE）
   - 疑点说明（说清楚"为什么怀疑"）
   - 验证步骤（分多步，能早停就早停）
   - 判定标准（表格形式，什么证据对应什么结论，避免开发机 AI 拍脑袋）
   - 修复方案（改或不改，改的话改哪里）

3. **2 项额外静态核对**：
   - ONNX 输入输出约定（防止最基础的 shape/dtype 不对齐）
   - UE 版本的 `PushLUT` 色彩空间行为（UE 5.2 之后 `CombineLUTs` 可能用 log 编码，这是我之前漏掉的一个潜在大坑）

4. **执行顺序 + 时间预算** — 明确 2~3 小时，超时就标"待人工确认"不要硬撑。

5. **回报格式** — 让开发机把结论**追加到同一文件的回报区**，训练机只需要 `git pull` 就能看到结果，不需要新文件、不需要额外协商。

## 几个设计取舍

**为什么不直接指示"把 `1.0f/PreExposure` 改成 `1.0f`"**：
因为这是个**未经验证的推测**。如果贸然指示改，开发机 AI 会照做，但如果推测错了就引入新 bug。所以我要求先查证 UE 源码 + 跑 log 取证，**有证据再改**。这符合第一性原理。

**为什么让开发机判定而不是训练机**：
P1（训练/推理 domain shift）的修复有 3 条路径，其中"方向 A 训练侧做 ACES 预处理"需要训练机改 `train.py`。但**要不要走方向 A 取决于开发机实测看到的分布差距大小**。所以工单让开发机把"方向选择回报给训练机"，由训练机拍板，而不是让开发机自己做训练侧的决定。

**P2/P3 标为低优先级并明确"不要深入"**：
避免开发机 AI 把时间花在次要问题上。先把 P0 和 P1 搞清楚，这两个才是"颜色不准"的主因嫌疑。
