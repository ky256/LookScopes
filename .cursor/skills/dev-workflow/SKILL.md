# 开发流程规范

## 核心原则

**架构先行，迭代交付，留痕可溯。**
每个模块先出设计，确认后再编码；小步快跑持续迭代；所有决策和变更留下时间线记录。

## 开发节奏

### 新模块开发

```
1. 设计  → 在 docs/ 下写模块设计文档（或更新 ARCHITECTURE.md 对应章节）
2. 确认  → 与用户讨论设计，对齐预期
3. 骨架  → 创建目录结构 + 接口定义（只有签名，无实现）
4. 实现  → 逐个方法填充实现，每完成一个可测试单元就验证
5. 集成  → 与已有模块联调
6. 留痕  → 更新 ARCHITECTURE.md + CHANGELOG.md + 必要时写 ADR
```

### 迭代改进

```
1. 明确问题 → 用户描述痛点或需求
2. 影响分析 → 评估改动范围，列出受影响文件
3. 方案讨论 → 如有多种方案则先讨论再动手
4. 实施     → 编码 + 自测
5. 留痕     → 同步 CHANGELOG + 如涉及架构变更则更新文档和 ADR
```

## 时间线管理（三层留痕）

### 第一层：版本变更日志 — CHANGELOG.md

记录"做了什么"。位于项目根目录 `CHANGELOG.md`。

**格式：**

```markdown
# Changelog

## [Unreleased]

### 新增
- [中控] 实现意图路由的 capability 匹配

### 变更
- [编排] 任务链支持并行步骤

### 修复
- [Agent:Houdini] 修复 FBX 导出时坐标系错误

---

## [0.1.0] - 2026-03-15

### 新增
- 项目初始化，创建基础目录结构
- ARCHITECTURE.md 架构设计文档 v1
```

**规则：**
- 遵循 [Keep a Changelog](https://keepachangelog.com) 格式
- 分类：新增 / 变更 / 修复 / 移除 / 废弃 / 安全
- 每条用 `[模块]` 前缀（复用 commit-conventions 的模块前缀）
- 版本号遵循语义化版本：`MAJOR.MINOR.PATCH`
  - MAJOR: 架构级破坏性变更
  - MINOR: 新模块 / 新能力上线
  - PATCH: 修复 / 小改进
- `[Unreleased]` 区持续累积，发版时转为具体版本号

### 第二层：架构决策记录 — ADR

记录"为什么这样决定"。位于 `docs/adr/` 目录。

**何时写 ADR：**
- 选择了技术方案（如 HTTP vs MCP）
- 放弃了某个方案（如不用 LangChain）
- 改变了之前的决定
- 引入新的第三方依赖

**文件命名：** `docs/adr/NNN-标题.md`，编号递增。

**模板：**

```markdown
# ADR-NNN: 简短标题

- 状态：已采纳 | 已废弃 | 已被 ADR-XXX 取代
- 日期：YYYY-MM-DD
- 关联：ARCHITECTURE.md 第X节

## 背景

什么问题需要决策？当时的约束和上下文。

## 方案

### 方案 A: ...
- 优势：...
- 劣势：...

### 方案 B: ...
- 优势：...
- 劣势：...

## 决定

选择了哪个方案，核心理由（一句话）。

## 后果

这个决定带来的影响，已知的取舍。
```

**ARCHITECTURE.md 中已有的决策备忘应逐步迁移为独立 ADR。**

### 第三层：开发日志 — devlog

记录"每天/每阶段发生了什么"。位于 `docs/devlog/` 目录。

**文件命名：** `docs/devlog/YYYY-MM-DD.md`，按日期。

**模板：**

```markdown
# 2026-03-13 开发日志

## 本次目标
- 完成 Phase 0 基础设施搭建

## 完成事项
- 创建项目结构
- 提取 universal 模块到 blueprint/
- ...

## 遇到的问题
- Houdini Agent 的 MCP 接口与 /chat 端点如何共存 → 见 ADR-002

## 下一步
- 给 Houdini Agent 添加 /chat endpoint

## 备注
- 用户偏好：先看小规模测试结果再全量执行
```

**规则：**
- 每次有意义的开发会话结束时更新（不必每天）
- 重点记录：完成了什么、卡在哪、下一步、涌现的偏好/规律
- 可以由 AI 辅助生成，用户确认后归档
- 这是项目级"记忆"的来源，中控 Agent 未来的 Unified Memory 将从这里冷启动

## 文档体系总览

```
docs/
├── ARCHITECTURE.md        ← 架构设计（活文档，持续更新）
├── adr/
│   ├── 001-通信选HTTP不选MCP.md
│   ├── 002-不引入LangChain.md
│   └── ...
└── devlog/
    ├── 2026-03-13.md
    ├── 2026-03-14.md
    └── ...

CHANGELOG.md               ← 项目根目录
```

| 文档 | 回答的问题 | 更新频率 |
|------|-----------|---------|
| ARCHITECTURE.md | 系统是什么样的？ | 架构变更时 |
| CHANGELOG.md | 做了什么？ | 每次提交 |
| ADR | 为什么这样决定？ | 每次重要决策 |
| devlog | 今天发生了什么？ | 每次开发会话 |

## 目录结构约定

严格遵循 ARCHITECTURE.md 第十一节定义的项目结构。新增文件前先确认它属于哪个模块目录。

## 质量门禁

每次提交前确保：
- 代码通过 linter 检查
- 新增公开接口有 docstring
- 不引入循环依赖
- 不硬编码配置值（用 config/ 管理）
- CHANGELOG.md 的 `[Unreleased]` 区已更新
