# 讨论与决策流程

## 三种模式，按场景切换

### 模式 A：第一性原理分析

**适用场景：** 核心架构决策、新概念引入、"为什么要这样做"类问题。

```
1. 回到本质  → 这个问题的根本目的是什么？
2. 拆解假设  → 当前方案隐含了哪些假设？哪些可能是错的？
3. 从零推导  → 如果抛开惯例，最简单有效的解法是什么？
4. 对照原则  → 是否符合项目五大原则（自然语言即协议、文件即交付物、
                黑盒封装、自举能力、记忆即身份）？
5. 得出结论  → 给出推荐方案及理由
```

### 模式 B：多方案对比

**适用场景：** 技术选型、存在多种可行路径、需要权衡取舍。

```
1. 列出方案  → 至少2个可行方案，包含"不做"选项（如适用）
2. 对比维度  → 用表格从复杂度、性能、可维护性、与现有架构契合度等维度对比
3. 推荐选择  → 明确推荐一个，说明理由
4. 用户决策  → 等待用户确认或提出偏好
5. 执行       → 按确认方案实施
```

### 模式 C：逐步确认执行

**适用场景：** 明确的实现任务、多步骤操作、需要过程可控。

```
1. 制定计划  → 列出具体步骤（编号清单）
2. 用户确认  → 展示计划，等待确认或调整
3. 逐步执行  → 每完成一步，简要汇报结果
4. 检查点    → 关键步骤后暂停，确认方向正确再继续
```

### 模式 D：需求澄清（Brainstorm）

适用于需求模糊、存在多种实现路径、或架构级变更的场景。

**流程：**

1. **复述理解**：用一句话概括你对需求的理解，让用户确认方向
2. **提出关键问题**：每轮最多 3 个问题，聚焦于：
   - 实现路径：有哪些可选方案？用户倾向哪种？
   - 范围边界：哪些在范围内？哪些明确不做？
   - 优先级：如果要分阶段，先做哪部分？
3. **迭代澄清**：根据用户回答继续追问或收窄范围
4. **退出条件**：用户说"开始做"/"够了"/"直接 plan"，或所有关键歧义已消除

**禁止事项：**
- 禁止在 Brainstorm 阶段做任何实现决策
- 禁止假设用户的意图（有疑问就问）
- 禁止一次问超过 3 个问题（信息过载）

**退出后流向**：进入模式 A/B/C 或直接进入 Plan 阶段。

## 模式选择指引

| 信号 | 推荐模式 |
|------|---------|
| 用户问"为什么"/"该不该" | A：第一性原理 |
| 用户问"用X还是Y" / 存在明显取舍 | B：多方案对比 |
| 用户说"帮我做X" / 任务明确 | C：逐步确认 |
| 大型功能 / 架构级变更 / 多种实现路径 | D：需求澄清（Brainstorm） |
| 用户说"我不确定" / 需求模糊 | 先 D 澄清需求，再按 A/B/C 决策 |
| 涉及架构变更 | A → B → C（完整链路） |

## 通用规则

- **不要假设。** 有疑问就问，不要自行脑补用户意图。
- **先理解再动手。** 复述理解，确认对齐后再执行。
- **保持透明。** 说明你在做什么、为什么这么做。
- **粒度适当。** 简单改动不需要走完整流程；复杂决策不要跳过讨论。
- **尊重已有设计。** ARCHITECTURE.md 是当前权威，改动前先讨论。
